Add a small guard in tick(): if a fetched dev_proposals row has both an empty/blank title and empty/blank proposal body, do not attempt to send a Telegram message. Instead update that row's status to 'invalid' (or another sentinel), set notified_at=datetime('now') and notified_msg_id='INVALID' so it won't be repeatedly retried. This is a low-risk single-file change: it prevents wasteful repeated sends/DB churn for malformed/empty proposals and makes the daemon more robust without changing external APIs or requiring schema changes beyond using existing status and fields.
