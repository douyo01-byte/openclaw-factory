Add a small guard after extracting chat_id and message_id to skip processing/upserting updates that lack a valid chat id or message id (>0). Currently message_id is coerced to 0 when missing which can cause repeated INSERT OR IGNORE collisions on the unique inbox_commands(chat_id, message_id) index and pollute the table with bogus records. Change: after computing chat_id and message_id, if not chat_id or message_id <= 0 then continue (skip this update). This is a minimal, low-risk change localized to the for upd in updates loop and prevents invalid rows from being written and later causing mapping/lookup confusion.
